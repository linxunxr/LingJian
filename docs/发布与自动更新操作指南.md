## 1. 整体流程

灵鉴的发布与更新闭环基于 **Tauri Updater + GitHub Releases + 腾讯云 COS + GitHub Actions**：

```
开发者在本地打 tag (v0.2.0) 并推送
        │
        ▼
GitHub Actions 自动触发 Release 工作流
        │
        ▼
多平台并行构建（Win / macOS / Linux）+ Ed25519 签名
        │
        ▼
自动创建 GitHub Release，附带：
  ├─ 各平台安装包（.exe / .dmg / .deb / .rpm / .AppImage）
  └─ latest.json（更新清单，含版本号 / 下载地址 / 签名）
        │
        ▼
同步到腾讯云 COS（国内加速）
  ├─ 重写 latest.json 的 URL 指向 COS
  └─ 上传所有安装包 + latest.json 到 cos
        │
        ▼
已安装旧版本的用户启动灵鉴
        │
        ▼
灵鉴读取 latest.json（COS 优先，GitHub 备份）
  → 对比版本 → 从 COS 下载安装包 → 签名验证 → 自动安装重启
```

**核心设计**：

- **双源容错**：更新源以腾讯云 COS 为主（国内毫秒级），GitHub Releases 为备份（COS 不可达时兜底）
- **签名验证**：每个安装包用 Ed25519 私钥签名，应用内置公钥校验，确保传输过程未被篡改
- **低维护**：CI 自动完成构建 → 签名 → 发布 → 同步 COS，无需人工介入
- **跨平台**：一次打 tag，四平台同时构建

## 2. 一次性配置：GitHub Secrets

签名密钥用于验证安装包未被篡改。密钥对已生成，私钥**必须**存入 GitHub Secrets。此步骤仅需做一次。

### 2.1 获取密钥值

项目初始化时已生成 Ed25519 密钥对：

| 项目 | 说明 | 保密级别 |
|------|------|----------|
| **公钥** | 已内置到 `tauri.conf.json` 的 `pubkey` 字段 | 可公开 |
| **私钥** | base64 长字符串，构建时通过环境变量注入 | ⚠️ 绝密 |
| **密码** | 私钥的保护密码 | ⚠️ 绝密 |

> 如需重新生成密钥对（例如私钥丢失），执行：
> ```bash
> npx tauri signer generate --password "<新密码>"
> ```
> 生成后需更新 `tauri.conf.json` 的 `pubkey`，并用新私钥重新发布。

### 2.2 添加 Secrets 到仓库

1. 打开 GitHub 仓库：`https://github.com/linxunxr/LingJian`
2. 进入 **Settings** → **Secrets and variables** → **Actions**
3. 点击 **New repository secret**，依次添加四个：

| Secret 名称 | 值 | 说明 |
|-------------|-----|------|
| `TAURI_SIGNING_PRIVATE_KEY` | 私钥值（整段 base64） | 构建时签名安装包 |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | 私钥密码 | 解锁私钥 |
| `COS_SECRET_ID` | 腾讯云 CAM 子账号 SecretId | CI 同步安装包到 COS |
| `COS_SECRET_KEY` | 腾讯云 CAM 子账号 SecretKey | CI 同步安装包到 COS |

> ⚠️ **私钥安全须知**：
> - 添加后只写不读，任何人都无法再次查看值
> - 请同时在本地密码管理器（如 1Password、Bitwarden）保存一份
> - **丢失私钥将无法发布可更新的版本**，用户只能手动重装新版
> - 建议每 12 个月轮换一次密钥

### 2.3 创建腾讯云 COS Bucket（更新镜像源）

自动更新采用**双源容错**：腾讯云 COS 优先（国内快）、GitHub 备份。需创建专用 Bucket：

1. 登录 [COS 控制台](https://console.cloud.tencent.com/cos)
2. 创建 Bucket：
   - **名称**：`lingjian-releases-1433733625`
   - **地域**：`ap-guangzhou`（广州）
   - **访问权限**：**公有读私有写**（安装包需公开下载）
   - **存储类型**：标准存储
3. 配置 CORS（允许应用跨域访问）：
   - 来源：`*`
   - 操作：`GET`
   - Headers：`*`
4. 创建 CAM 子账号（或复用日志系统的子账号），授予该 Bucket 的**读写权限**
5. 将子账号的 SecretId / SecretKey 填入上面表格的 `COS_SECRET_ID` / `COS_SECRET_KEY`

> 💡 此 Bucket 与日志 Bucket（`path-of-idle-immortals-1433733625`）隔离，互不影响。
> 标准存储安装包约 8MB × 几个版本，月费 < 0.1 元。

## 3. 日常发布操作

配置完成后，每次发布新版本只需 3 步。

### 3.1 更新版本号

以下两个文件的 `version` 字段**必须保持一致**：

```json
// package.json
{
  "version": "0.2.0"   // ← 修改这里
}
```

```json
// src-tauri/tauri.conf.json
{
  "version": "0.2.0"   // ← 修改这里（与 package.json 一致）
}
```

> 版本号遵循 [语义化版本](https://semver.org/lang/zh-CN/)：`主版本.次版本.修订号`
> - 修订号（0.1.**0** → 0.1.**1**）：bug 修复
> - 次版本（0.**1**.0 → 0.**2**.0）：新功能，向下兼容
> - 主版本（**0**.1.0 → **1**.0.0）：重大变更

### 3.2 提交并打 Tag

```bash
# 1. 提交版本号变更
git add -A
git commit -m "chore: 发布 v0.2.0"

# 2. 打 tag（tag 名必须以 v 开头，触发工作流）
git tag v0.2.0

# 3. 推送 tag 到远程
git push origin v0.2.0
```

> 💡 也可同时推送代码和 tag：`git push origin develop && git push origin v0.2.0`

### 3.3 监控构建过程

推送 tag 后：

1. 进入仓库 **Actions** 页面，可看到 `Release` 工作流正在运行
2. 构建矩阵（4 个平台并行）：

| 任务 | 平台 | 产物 |
|------|------|------|
| windows-latest | Windows x64 | `.exe`（NSIS 安装包）|
| macos-latest (aarch64) | macOS Apple Silicon | `.dmg` |
| macos-latest (x86_64) | macOS Intel | `.dmg` |
| ubuntu-22.04 | Linux x64 | `.deb` / `.rpm` / `.AppImage` |

3. 构建约需 **10-20 分钟**（取决于 GitHub Actions 负载）
4. 全部成功后，自动在 **Releases** 页面创建发布

### 3.4 发布产物说明

构建完成后，GitHub Release 会包含：

| 文件 | 用途 |
|------|------|
| `灵鉴_0.2.0_x64-setup.exe` | Windows 安装包（支持自定义安装路径）|
| `灵鉴_0.2.0_aarch64.dmg` | macOS Apple Silicon 安装镜像 |
| `灵鉴_0.2.0_amd64.deb` | Linux Debian/Ubuntu 安装包 |
| `灵鉴_0.2.0_amd64.AppImage` | Linux 免安装版 |
| `灵鉴_0.2.0_amd64.rpm` | Linux Fedora/CentOS 安装包 |
| **`latest.json`** | **更新清单**（自动更新核心文件）|

`latest.json` 内容示例：

```json
{
  "version": "0.2.0",
  "notes": "灵鉴 0.2.0 更新内容",
  "pub_date": "2026-06-19T10:00:00Z",
  "platforms": {
    "windows-x86_64": {
      "signature": "...（Ed25519 签名）",
      "url": "https://github.com/linxunxr/LingJian/releases/download/v0.2.0/灵鉴_0.2.0_x64-setup.exe"
    }
  }
}
```

## 4. 验证自动更新

发布完成后，用旧版本验证自动更新是否生效。

### 4.1 安装旧版本

从 GitHub Releases 下载并安装一个旧版本（如 v0.1.0）。

### 4.2 触发更新检测

灵鉴有两种更新检测方式：

| 方式 | 触发时机 | 用户感知 |
|------|----------|----------|
| **自动检查** | 应用启动时静默执行 | 导航栏出现绿色闪烁「新版本」提示 |
| **手动检查** | 设置 → 应用更新 → 检查更新 | 卡片显示检查结果 |

### 4.3 完整更新流程

1. 启动灵鉴，等待几秒（启动时静默检查）
2. 导航栏右侧出现绿色闪烁的「新版本」提示
3. 点击提示，或进入 **设置 → 应用更新**
4. 卡片显示新版本号、发布日期、更新说明
5. 点击「下载并安装」
6. 进度条显示下载进度（0% → 100%）
7. 下载完成 → 签名验证 → 自动安装 → 重启
8. 重启后版本号已是新版（检查设置页确认）

## 5. 配置文件说明

以下配置已在代码中完成，**无需修改**，仅供参考理解。

### 5.1 `src-tauri/tauri.conf.json` — 更新源配置（双源容错）

```json
{
  "bundle": {
    "createUpdaterArtifacts": true
  },
  "plugins": {
    "updater": {
      "active": true,
      "endpoints": [
        "https://lingjian-releases-1433733625.cos.ap-guangzhou.myqcloud.com/latest.json",
        "https://github.com/linxunxr/LingJian/releases/latest/download/latest.json"
      ],
      "pubkey": "dW50cnVzdGVk...（Ed25519 公钥）",
      "windows": {
        "installMode": "passive"
      }
    }
  }
}
```

| 字段 | 作用 |
|------|------|
| `createUpdaterArtifacts` | 构建时生成签名安装包（`.exe` 对应 `-setup.exe`）|
| `endpoints[0]` | **腾讯云 COS**（国内优先，毫秒级响应）|
| `endpoints[1]` | **GitHub Releases**（备份兜底，COS 不可达时使用）|
| `pubkey` | 内置公钥，用于验证下载安装包的签名 |
| `windows.installMode` | Windows 安装模式：`passive`（静默安装带进度，无需用户交互）|

> Tauri updater 按数组顺序尝试 endpoints，第一个成功即用。COS 优先保证国内速度，GitHub 备份保证可用性。

### 5.2 `.github/workflows/release.yml` — 发布工作流（构建 + COS 同步）

工作流分两个 job：

**job 1: `publish`** — 多平台并行构建

```yaml
jobs:
  publish:
    strategy:
      matrix:
        include:
          - platform: windows-latest
            args: '--bundles nsis'    # Windows 只生成 NSIS exe
          # ... macOS / Linux
    steps:
      # ... 构建 + 签名 + 上传到 GitHub Release
```

**job 2: `mirror-to-cos`** — 等所有平台构建完成后，同步到腾讯云 COS

```yaml
  mirror-to-cos:
    needs: publish          # 等构建完成
    steps:
      - name: 下载 Release 所有 assets
        run: gh release download ... --dir dist
      - name: 重写 latest.json 的 URL 指向 COS
        run: node scripts/rewrite-latest-json.mjs dist/latest.json
      - name: 上传到 COS
        run: coscmd upload "$file" "/$filename"
```

| 步骤 | 作用 |
|------|------|
| 下载 assets | 用 `gh release download` 拉取本次 Release 全部产物 |
| 重写 latest.json | `scripts/rewrite-latest-json.mjs` 把下载 URL 从 GitHub 改为 COS |
| 上传 COS | 用 `coscmd` 上传安装包 + 改写后的 latest.json |

> **为什么要重写 latest.json？** tauri-action 生成的 latest.json 中 URL 指向 GitHub，但用户应从 COS 下载（国内快）。脚本把每个平台的 URL 改为 `https://lingjian-releases-...cos.../文件名`，并上传同名安装包到 COS。

| 设计点 | 说明 |
|--------|------|
| 触发条件 | 推送 `v*` 格式的 tag |
| 构建矩阵 | 4 平台并行，互不阻塞 |
| Windows | 用 `--bundles nsis` 只生成 exe（支持自定义安装路径，便携存储）|
| 签名注入 | Secrets 自动转为构建环境变量 |
| Release 自动化 | tauri-action 自动创建 Release 并上传产物 + latest.json |

## 6. 常见问题

### Q1：第一次发布怎么操作？

第一次发布没有"旧版本"可以更新。直接发布 v0.1.0 作为初始版本供用户安装。从 v0.2.0 开始，v0.1.0 的用户才能收到自动更新。

### Q2：构建失败了怎么办？

进入 **Actions** 页面，点击失败的工作流查看日志。常见原因：

| 现象 | 原因 | 解决 |
|------|------|------|
| 签名相关错误 | Secrets 未配置或值错误 | 重新检查第 2 步配置 |
| 版本号错误 | package.json 与 tauri.conf.json 不一致 | 同步两个文件的 version |
| Rust 编译失败 | 代码有编译错误 | 本地 `cargo build` 确认通过后再发布 |
| 网络超时 | GitHub Actions 负载高 | 重新运行失败的任务（Re-run jobs）|
| macOS 签名问题 | 无 Apple 开发者证书 | 当前未配置 macOS 公证，用户首次打开需在「系统设置 → 隐私与安全」允许 |

### Q3：能否发布预发布版（pre-release）？

当前工作流配置为 `prerelease: false`（正式发布）。如需预发布版：

**方式一**：修改 `release.yml`：
```yaml
with:
  prerelease: true   # 改为 true
```

**方式二**：发布后在 GitHub Release 页面手动勾选「Set as a pre-release」。

> 预发布版不会被 `latest.json` 指向（latest 始终指向最新的正式 Release），用户不会自动更新到预发布版。

### Q4：如何更新已发布版本的说明？

发布后可在 GitHub Release 页面编辑 `releaseBody`（更新说明），但 **latest.json 的内容在发布时已固定**。若需修改更新说明供用户看到，需发布新版本。

### Q5：如何取消一次发布？

```bash
# 1. 删除 GitHub Release（在 Releases 页面点击 Delete）
# 2. 删除本地和远程 tag
git tag -d v0.2.0
git push origin :refs/tags/v0.2.0
```

> ⚠️ 注意：**已更新到该版本的用户无法回退**，除非他们手动下载旧版重装。

### Q6：私钥丢了怎么办？

**无法补救**。必须：

1. 重新生成密钥对：`npx tauri signer generate --password "<新密码>"`
2. 更新 `tauri.conf.json` 的 `pubkey` 为新公钥
3. 更新 GitHub Secrets 为新私钥和密码
4. 发布新版本

旧版本用户因公钥不匹配无法自动更新，只能手动下载新版重装。**所以务必妥善保存私钥和密码。**

### Q7：macOS 用户提示「无法验证开发者」？

当前未配置 Apple Developer 证书和公证（notarization）。macOS 用户首次打开需：

1. 双击 .dmg 拖入 Applications
2. 首次启动若提示无法打开，进入 **系统设置 → 隐私与安全性**
3. 点击「仍要打开」确认

如需消除此提示，需配置 Apple 开发者证书（每年 $99），可后续在 release.yml 中加入公证步骤。

### Q8：能否本地构建签名版本测试？

可以。设置环境变量后本地构建：

```bash
# Windows (cmd)
set TAURI_SIGNING_PRIVATE_KEY=<私钥>
set TAURI_SIGNING_PRIVATE_KEY_PASSWORD=<密码>
npm run tauri build

# 或 PowerShell / macOS / Linux
$env:TAURI_SIGNING_PRIVATE_KEY="<私钥>"
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD="<密码>"
npm run tauri build
```

本地构建也会生成签名安装包，但不会上传到 GitHub Release，不会触发自动更新。
