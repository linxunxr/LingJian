## 2. 系统概述

### 2.1 背景

项目已有自研日志系统（`src/utils/common/logger.ts`），支持 DEBUG/INFO/WARN/ERROR 多级别日志，内存保存最多 1000 条，并提供 `getLogs()`、`getRecentLogs()`、`exportLogsAsJson()` 等导出接口。

当前日志仅存在于客户端内存中，当用户遇到问题时，开发团队无法获取日志进行排查。本方案设计一套从客户端到服务端的完整日志上报链路，实现：

- 玩家手动点击按钮即可将日志上报至服务端
- 客户端将日志打包为单个 gzip 压缩包上传（一次反馈 = 一个文件）
- 服务端生成 reportId，压缩包存入 COS，自动创建 GitHub Issue
- 分析工具从 Issue 解析 reportId，通过 SCF 下载压缩包，本地解压分析

### 2.3 设计原则

| 原则 | 说明 |
|------|------|
| 最小侵入 | 新增 `logReporter` 模块复用现有 `logger.getLogs()` 接口，不改动 Logger 核心逻辑 |
| 用户主动触发 | 仅用户手动点击"问题反馈"按钮时上报，不静默上传 |
| 一次一包 | 每次反馈打包为一个 gzip 压缩文件，简单可追溯 |
| 安全传输 | HMAC-SHA256 签名 + 时间戳防重放，敏感字段脱敏 |
| 可靠交付 | 失败自动重试最多 3 次，本地保留上报历史 |
| 频率限制 | 客户端 + 服务端双重频率限制，24h 内最多 5 次 |
| 可追溯 | 每次上报生成唯一 reportId（UUID），贯穿 COS → Issue → 分析工具全链路 |

### 3.2 上传格式

客户端将日志打包为单个 gzip 压缩文件上传，通过 HTTP Header 传递元信息。

```
POST /report/log
Headers:
  Content-Type: application/gzip
  Content-Encoding: gzip
  X-App-Version: 1.2.3
  X-Platform: electron
  X-Realm: 筑基期
  X-Play-Time: 86400                 ← 秒
  X-User-Description: <base64>       ← 用户问题描述（base64，避免中文乱码）
  X-Timestamp: 1716800000000
  X-Signature: <hex-hmac-sha256>     ← HMAC(compressed_body + ':' + timestamp, secret)
```

> **设计意图**：元信息（版本、平台、描述等）放在 Header 中，SCF 无需解压即可获取这些信息用于生成 Issue 摘要和命名 COS 对象。日志内容本身作为 gzip body 透传，减少上传体积。

### 3.3 核心逻辑

```typescript
// src/utils/common/logReporter.ts
import { logger } from './logger'
import { LogLevel } from './logger'

const REPORT_CONFIG = {
  maxRecentLogs: 500,         // 收集最近 500 条
  maxRetries: 3,              // 失败重试 3 次
  rateLimitMax: 5,            // 24h 最多上报 5 次
  rateLimitWindow: 86400000,  // 24h 毫秒
}

/**
 * 收集全部日志，打包为 gzip 二进制
 */
async function buildReportBlob(userDescription: string): Promise<Blob> {
  const payload = {
    exportedAt: new Date().toISOString(),
    logs: logger.getRecentLogs(REPORT_CONFIG.maxRecentLogs),
  }
  const json = JSON.stringify(sanitize(payload))
  // 使用浏览器原生 CompressionStream API
  const stream = new Blob([json])
    .stream()
    .pipeThrough(new CompressionStream('gzip'))
  return new Response(stream).blob()
}
```

### 3.4 签名（对压缩后的二进制签名）

与 v1.2 的区别：签名对象从 JSON 字符串变为 gzip 压缩后的二进制数据。

```typescript
async function signBlob(blob: Blob, timestamp: number): Promise<string> {
  const secret = import.meta.env.VITE_LOG_REPORT_SECRET
  const data = await blob.arrayBuffer()
  // HMAC 输入 = 压缩后的二进制 + ':' + 时间戳
  const encoder = new TextEncoder()
  const key = await crypto.subtle.importKey(
    'raw', encoder.encode(secret),
    { name: 'HMAC', hash: 'SHA-256' },
    false, ['sign']
  )
  const suffix = encoder.encode(':' + timestamp)
  const combined = new Uint8Array(data.byteLength + suffix.byteLength)
  combined.set(new Uint8Array(data), 0)
  combined.set(suffix, data.byteLength)
  const sig = await crypto.subtle.sign('HMAC', key, combined)
  return Array.from(new Uint8Array(sig)).map(b => b.toString(16).padStart(2, '0')).join('')
}
```

### 3.5 上报函数

```typescript
export async function uploadLogs(userDescription: string): Promise<ReportResponse> {
  if (!canReport()) {
    logger.warn('日志上报', '已达到24小时上报上限')
    return { success: false, error: '频率限制：24小时内最多上报5次' }
  }

  const blob = await buildReportBlob(userDescription)
  const timestamp = Date.now()

  for (let attempt = 1; attempt <= REPORT_CONFIG.maxRetries; attempt++) {
    try {
      const signature = await signBlob(blob, timestamp)

      const resp = await fetch(import.meta.env.VITE_LOG_REPORT_URL, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/gzip',
          'X-App-Version': __APP_VERSION__,
          'X-Platform': window.electron ? 'electron' : 'web',
          'X-Realm': getCurrentRealm(),
          'X-Play-Time': String(getPlayTimeSeconds()),
          'X-User-Description': btoa(unescape(encodeURIComponent(userDescription))),
          'X-Timestamp': String(timestamp),
          'X-Signature': signature,
        },
        body: blob,
      })

      if (resp.ok) {
        recordReport()
        logger.info('日志上报', `上报成功，第${attempt}次尝试`)
        return await resp.json()
      }

      logger.warn('日志上报', `上报失败(${resp.status})，第${attempt}次尝试`)
    } catch (e) {
      logger.error('日志上报', `网络错误，第${attempt}次尝试`, e)
    }
  }

  logger.error('日志上报', '重试3次后仍失败')
  return { success: false, error: '网络错误，请稍后重试' }
}
```

> **注意**：由于 API 网关对 binary body 有尺寸限制（通常 6MB），若压缩后仍超限，需降级分片上传或提示用户缩减日志量。实际 500 条日志 gzip 后约 50-200KB，远低于限制。

### 3.7 敏感字段脱敏

在上报前递归扫描日志数据中的敏感字段：

```typescript
const SENSITIVE_PATTERNS = [/token/i, /key/i, /secret/i, /password/i, /auth/i]

function sanitize(obj: unknown): unknown {
  if (typeof obj !== 'object' || obj === null) return obj
  if (Array.isArray(obj)) return obj.map(item => sanitize(item))
  return Object.fromEntries(
    Object.entries(obj as Record<string, unknown>).map(([k, v]) => [
      k,
      SENSITIVE_PATTERNS.some(p => p.test(k)) ? '[REDACTED]' : sanitize(v),
    ])
  )
}
```

### 3.8 频率限制

```typescript
function canReport(): boolean {
  const records: number[] = JSON.parse(
    localStorage.getItem('log_report_records') || '[]'
  )
  const now = Date.now()
  const recent = records.filter(t => now - t < REPORT_CONFIG.rateLimitWindow)
  return recent.length < REPORT_CONFIG.rateLimitMax
}

function recordReport(): void {
  const records: number[] = JSON.parse(
    localStorage.getItem('log_report_records') || '[]'
  )
  records.push(Date.now())
  localStorage.setItem('log_report_records', JSON.stringify(records))
}
```

## 4. 创建方式选型分析

选择 **容器镜像**（WebServer 镜像 + Web 函数）方式，核心考量是跨云可迁移性——Docker 镜像可在 AWS Lambda、Google Cloud Run、阿里云函数计算、自建 K8s 等任意平台运行，不被腾讯云锁定。

### 4.1 容器镜像方式深入分析

容器镜像方式下，存在**两个独立维度**需要区分：**函数类型** 和 **镜像类型**。很多人将这两个维度混淆，导致选型错误。

#### 4.1.1 维度一：函数类型（决定 SCF 如何调用你的函数）

| 函数类型 | 触发机制 | 事件格式 | 典型入口 |
|----------|---------|---------|---------|
| **事件函数** (Event) | 事件触发，API 网关将请求转为 JSON 事件 | `{ body, headers, ... }` JSON | `main_handler(event, context)` |
| **Web 函数** (Web) | HTTP 请求直接触发，原生 HTTP 语义 | 原始 HTTP Request/Response | Express/Fastify 等标准 Web 框架 |

**事件函数**：API 网关收到 HTTP 请求后，将其元信息（headers, body, path 等）封装为 JSON 对象传给函数。函数通过解析 `event.body` 获取请求体。

**Web 函数**：专为 Web 服务场景优化。API 网关作为七层反向代理，将 HTTP 请求**直接透传**到函数实例。函数可以像普通 Web 服务器一样处理请求（原生 header、cookie、content-type 等），无需解析 event 结构。

**关键约束**：Web 函数只支持 HTTP 触发，不支持定时器、COS 事件等其他触发器。

#### 4.1.2 维度二：镜像类型（仅容器镜像方式下需要选择）

| 镜像类型 | 是否需 HTTP Server | 端口暴露 | 事件注入方式 | 实例生命周期 |
|----------|-------------------|---------|-------------|-------------|
| **WebServer 镜像** | ✅ 必须内嵌 HTTP Server | 固定监听 **9000** 端口 | HTTP Request body | 常驻，处理完请求后保留实例 |
| **Job 镜像** | ❌ 不需要 | 不暴露任何端口 | 环境变量 `SCF_CUSTOM_CONTAINER_EVENT` | 执行完 CMD 后**自动销毁** |

**WebServer 镜像**：SCF 拉起容器后，执行指定的 `Command` + `Args` 启动 HTTP Server，固定监听 **9000 端口**。所有调用请求（事件触发或 HTTP 触发）均以 HTTP 请求形式传入该 Server：
- 事件函数：需监听 `/event-invoke` 固定路径
- Web 函数：无需指定路径，API 网关七层反向代理直接透传

**Job 镜像**：容器内无需 HTTP Server。SCF 拉取镜像后执行 `CMD`/`EntryPoint` 定义的命令，事件体通过环境变量 `SCF_CUSTOM_CONTAINER_EVENT` 注入。命令执行完毕后，实例**自动销毁**。设计初衷：让已有容器化批处理任务无需改造即可迁移到 SCF。

**Job 镜像的关键限制**：

| 限制项 | 值 | 对日志上报的影响 |
|--------|-----|-----------------|
| 单环境变量上限 | **128KB** | 日志 payload 通常远超此值 |
| 所有环境变量总上限 | **2MB** | 500 条日志 + 上下文信息容易接近上限 |
| 请求响应模式 | **异步** | 无法直接向客户端返回处理结果 |
| 仅支持事件函数 | 不能选 Web 函数 | HTTP 语义丢失，需手动解析 event |
| 不暴露端口 | 无法透传 HTTP | 无法用标准 Web 框架处理请求 |

#### 4.1.3 四种组合兼容性矩阵

|  | WebServer 镜像 | Job 镜像 |
|--|---------------|---------|
| **事件函数** | ✅ 支持。监听 `/event-invoke`，event body 为 HTTP POST body | ✅ 支持。事件体通过环境变量注入，执行完销毁 |
| **Web 函数** | ✅ 支持。API 网关直接透传 HTTP 请求 | ❌ **不支持**。Job 镜像不暴露端口，无法透传 HTTP |

#### 4.1.4 本场景分析

日志上报的核心需求：
- 接收客户端 POST 的 JSON body（~50KB-500KB）
- 同步返回处理结果（Issue 链接）
- 使用标准 HTTP 语义处理请求

逐方案分析：

| 方案 | 可行性 | 理由 |
|------|--------|------|
| WebServer + Web 函数 | ✅✅ **最佳** | 原生 HTTP 透传，标准 Web 框架，同步响应 |
| WebServer + 事件函数 | ✅ 可行 | 需监听 `/event-invoke`，手动解析 event body |
| Job + 事件函数 | ❌ **不可行** | 128KB 环境变量限制，且异步模式无法返回结果 |
| Job + Web 函数 | ❌ 不支持 | SCF 不支持此组合 |

### 5.3 SCF 核心代码骨架

```javascript
// scf/index.js
const crypto = require('crypto')
const express = require('express')
const { uploadToCOS, getCOSObject } = require('./cos')
const { createGithubIssue } = require('./github')

const app = express()

// 接收原始 gzip 二进制 body（不做 JSON 解析）
app.use(express.raw({ type: 'application/gzip', limit: '10mb' }))

app.post('/*', async (req, res) => {
  // 1. 时间戳防重放
  const ts = parseInt(req.headers['x-timestamp'], 10)
  if (!ts || Math.abs(Date.now() - ts) > (process.env.TIMESTAMP_TOLERANCE || 300) * 1000) {
    return res.status(401).json({ success: false, error: 'timestamp expired' })
  }

  // 2. 读取原始 gzip 二进制 body
  const rawBody = req.body  // Buffer
  if (!Buffer.isBuffer(rawBody) || rawBody.length === 0) {
    return res.status(400).json({ success: false, error: 'empty body' })
  }

  // 3. HMAC 验签（对 gzip 二进制 + 时间戳签名）
  const expected = hmacSha256(rawBody, ts, process.env.REPORT_SECRET)
  if (req.headers['x-signature'] !== expected) {
    return res.status(401).json({ success: false, error: 'signature mismatch' })
  }

  // 4. 频率限制（服务端兜底）
  const clientIp = req.ip || req.headers['x-forwarded-for']
  if (!rateLimitCheck(clientIp)) {
    return res.status(429).json({ success: false, error: 'rate limit exceeded' })
  }

  // 5. 从 Header 提取元信息（无需解压）
  const meta = {
    appVersion: req.headers['x-app-version'] || 'unknown',
    platform: req.headers['x-platform'] || 'unknown',
    realm: req.headers['x-realm'] || 'unknown',
    playTime: req.headers['x-play-time'] || '0',
    userDescription: decodeBase64Header(req.headers['x-user-description']),
    reportTime: new Date(ts).toISOString(),
  }

  // 6. 生成唯一 reportId（UUID v4）
  const reportId = generateUUID()

  try {
    // 7. 并行：存 COS（核心）+ 创建 Issue
    const [, issue] = await Promise.all([
      uploadToCOS(reportId, rawBody, meta),
      createGithubIssue(reportId, meta),
    ])


    res.json({ success: true, reportId, issueUrl: issue.html_url })
  } catch (err) {
    console.error('上报处理失败:', err)
    res.status(500).json({ success: false, error: err.message })
  }
})

// 下载端点：从 COS 直取 gzip 文件
app.get('/logs/:reportId', async (req, res) => {
  const { reportId } = req.params
  const apiKey = req.headers['x-api-key']
  if (!apiKey || apiKey !== process.env.LOG_DOWNLOAD_API_KEY) {
    return res.status(401).json({ error: 'unauthorized' })
  }
  try {
    const key = `logs/${reportId}.gz`
    const { body, headers: cosHeaders } = await getCOSObject(key)
    res.set({
      'Content-Type': 'application/gzip',
      'Content-Disposition': `attachment; filename="${reportId}.gz"`,
      'X-Report-Time': cosHeaders['x-cos-meta-report-time'] || '',
      'X-App-Version': cosHeaders['x-cos-meta-app-version'] || '',
    })
    res.send(body)
  } catch (err) {
    if (err.statusCode === 404) {
      return res.status(404).json({ error: 'report not found' })
    }
    res.status(500).json({ error: err.message })
  }
})

app.listen(9000, () => console.log('Log reporter listening on port 9000'))

// 签名工具：对二进制 Buffer + 时间戳做 HMAC
function hmacSha256(buf, ts, secret) {
  const hmac = crypto.createHmac('sha256', secret)
  hmac.update(buf)
  hmac.update(':' + ts)
  return hmac.digest('hex')
}

function decodeBase64Header(val) {
  if (!val) return ''
  try {
    return Buffer.from(val, 'base64').toString('utf-8')
  } catch {
    return ''
  }
}
```

### 5.4 COS 存储（核心）

压缩包直接存入 COS，无需解压。每条上报 = 一个 `.gz` 文件，命名规则为 `logs/{reportId}.gz`。

#### 5.4.1 存储类型选型

腾讯云 COS 提供多种存储类型，对比如下（广州地域刊例价）：

| 存储类型 | 单价（元/GB/月） | 最短存储 | 数据取回 | 读取延迟 | 本场景评估 |
|---------|-----------------|---------|---------|---------|-----------|
| **标准存储 STANDARD** | **0.099** | 无限制 | 免费 | 毫秒级 | ✅ 推荐 |
| 低频存储 STANDARD_IA | 0.08 | 30 天 | 收费 | 毫秒级 | ❌ 最短存储限制 |
| 归档存储 ARCHIVE | 0.033 | 90 天 | 收费 | 需解冻 1-5 分钟 | ❌ 不能即时下载 |
| 深度归档 DEEP_ARCHIVE | ~0.01 | 180 天 | 收费 | 需解冻 12-48h | ❌ 不可接受 |
| 智能分层 INTELLIGENT_TIERING | 自动切换 | 无限制 | 按层计费 | 热层毫秒级 | ❌ 收监控费 |
| 标准存储（多 AZ） | 0.15 | 无限制 | 免费 | 毫秒级 | ❌ 同城容灾无用，贵 50% |

逐个排除的原因：

- **低频存储**：最短 30 天，Issue 处理完就删却要付满 30 天。每次下载收数据取回费
- **归档/深度归档**：下载前需"解冻"（1 分钟到 48 小时），开发者排查不能等
- **智能分层**：每个对象收监控费，文件数多时成本不可控。且 <64KB 的文件常驻热层，实际不会降冷
- **多 AZ**：贵 50% 换来同城容灾，日志 60 天后就删了，冗余无意义

**选择标准存储 STANDARD**。单次上报 gzip 后约 50-200KB，按月均 100 次上报，总存储约 15MB，月费不到 1 分钱。真正的成本优化不在存储类型，而在生命周期规则自动过期删除（见 §7.1）。

#### 5.4.2 免费额度

企业用户开通 COS 后赠送 **标准存储容量包 1TB，有效期 180 天**（不能抵扣流量、请求等费用）。资源包到期或用尽后，系统自动切换为按量计费，存储桶数据不会被删除，无须迁移。

对日志上报场景，1TB 足够存储约 50 亿条上报的日志包，180 天内完全免费。

#### 5.4.3 COS 代码实现

```javascript
// scf/cos.js
const COS = require('cos-nodejs-sdk-v5')

const cos = new COS({
  SecretId: process.env.COS_SECRET_ID,
  SecretKey: process.env.COS_SECRET_KEY,
})

async function uploadToCOS(reportId, gzipBuffer, meta) {
  const key = `logs/${reportId}.gz`
  await cos.putObject({
    Bucket: process.env.COS_BUCKET,
    Region: process.env.COS_REGION,
    Key: key,
    Body: gzipBuffer,
    ContentType: 'application/gzip',
    Metadata: {
      'app-version': meta.appVersion,
      'platform': meta.platform,
      'report-time': meta.reportTime,
    },
  })
  return key
}

async function getCOSObject(key) {
  return cos.getObject({
    Bucket: process.env.COS_BUCKET,
    Region: process.env.COS_REGION,
    Key: key,
  })
}
```

> **COS 优势**：gzip 二进制直存，无格式转换开销；标准存储毫秒级读取；天然支持大文件（10MB+）；1TB/180 天免费。

## 6. 安全设计

### 6.1 防重放攻击

- 客户端每次请求携带 `X-Timestamp`（毫秒时间戳）
- SCF 校验该时间戳与服务器时间差不超过 **5 分钟**（可配置）
- HMAC 签名计算包含时间戳：`HMAC(body + ':' + timestamp)`，篡改时间戳会导致签名不匹配

### 6.2 密钥管理

- 客户端密钥通过 `VITE_LOG_REPORT_SECRET` 环境变量注入，构建时打包
- SCF 密钥存储在云函数环境变量中，不提交到代码仓库
- 建议每 3 个月轮换密钥

### 6.3 脱敏策略

- 客户端上报前递归扫描 data 字段，匹配 `/token|key|secret|password|auth/i` 模式的键名替换为 `[REDACTED]`
- 日志中不包含玩家昵称、本地文件路径（在现有 logger 的 `globalErrorHandler` 中已做部分处理）

### 6.4 GitHub Token 最小权限

SCF 使用的 GitHub PAT 仅需以下最小权限：

| 权限 | 范围 | 用途 |
|------|------|------|
| `issues: write` | 目标仓库 | 创建 Issue 并添加 Labels |

不需要 `repo`（代码读写）、`pull_requests` 等权限。
